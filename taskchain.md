- **Next objective:** Contain and classify the XYZ / PhantomBlock defensive firmware-assessment prototype, determine whether it should become a dedicated Repository `0` host-observation adapter, merge an approved subset with JusticeForMe, or be retired, and approve its Repository `0` / Repository `1` gluing contracts before additional capability, packaging, publication, or deployment work proceeds.

- **MVP scope:** preserve current source and history; inventory implemented collectors, firmware/hash checks, kernel and management-channel heuristics, PCAP analysis, dashboard/API, extension interface, dry-run isolation abstraction, image/packaging scripts, tests, CI, documentation, and compliance mappings; classify the prototype against Repository `0`, Repository `1`, JusticeForMe, Bridge, review interfaces, and existing security repositories; define device, baseline, evidence, canonical field, deduplication, conflict, proposal, capability, receipt, revocation, temporal, privacy, incident, and rollback boundaries; select a dedicated destination, consolidation path, or retirement path; choose a license; document supported platforms and false-positive limits; establish pairwise and triple-overlap fixtures; and prohibit release or deployment until evidence and approvals are complete.
- **Priority:** Portfolio priority remains `P4 — HOLDING / INCUBATION`; this does not supersede QSO-GENOMES acceptance or any existing repository P0. A local containment task is `P0` because implementation and publication-capable automation were added before the approved incubation contract, creating a documented scope conflict.
- **Success criteria:** the prototype has one approved owner and destination, an approved JusticeForMe consolidation decision, or a recorded retirement decision; every claimed capability is classified as implemented code, configured workflow, locally tested behavior, unverified proposal, or accepted evidence; product and schema identities are reconciled; Repository `0` / Repository `1` responsibilities and JusticeForMe overlap are resolved; no license, data-rights, credential, firmware, hardware, networking, privacy, temporal, incident, or publication ambiguity remains; the Pages and release posture is explicitly approved or disabled; representative and adversarial tests quantify supported platforms and limitations; duplicate, conflict, partial, stale, replay, wrong-device, privacy, and response-boundary fixtures preserve identity and fail closed; reproducible exact-head evidence is retained; and no operational, certification, DoD/Army authorization, or detection-assurance claim exceeds the evidence.
- **Non-goals:** treating `Misc` as the permanent production repository; continuing feature expansion while ownership is unresolved; scanning systems without explicit authorization; exploiting management interfaces; flashing firmware; enabling disruptive isolation by default; storing customer data, credentials, firmware images, or sensitive findings; claiming comprehensive implant detection, CMMC status, STIG approval, Army authorization, or an ATO; allowing duplicate observations to masquerade as independent corroboration; allowing a report to self-authorize remediation; or allowing documentation publication to imply product release.
- **Release rationale:** Preserving the prototype while freezing promotion protects useful research evidence without rewarding a bypass of the repository intake contract. A dedicated, approved owner or explicit consolidation/retirement decision and compatible portfolio contracts are required before a privileged defensive assessment tool can be packaged, publicly represented, deployed, or integrated.

## Builder rules

Do not expand XYZ or add another experiment until Local P0 and Local P1 are resolved. Preserve all existing code and evidence.
